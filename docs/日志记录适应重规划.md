# 任务日志重构设计方案（简化版）

## 1. 问题背景

当前项目增加了"按自然语言改写"、"重试规划"的功能,并在确认规划正确后点击"确认执行"再继续执行。这导致:

- **Workflow文件**: 包含完整的规划历史(在`user_input_messages`数组中)
- **Task日志文件**: 每次执行生成一个独立的日志文件,同一workflow的多次执行被分割成多个文件

### 当前问题

1. 同一个workflow的多次执行被分割成多个task_log文件,难以追踪完整的执行流程
2. 无法区分执行阶段:初次规划、重规划、确认执行
3. 无法快速识别执行状态:执行成功/失败、规划成功/失败
4. 前端无法直观展示任务的完整生命周期

## 2. 设计目标

### 核心需求

1. **区分执行阶段**: 能够识别每次执行所处的阶段
   - `initial_planning`: 初次规划
   - `re_planning`: 重规划（合并改写规划和重试规划）
   - `execution`: 确认执行（开始执行任务）

2. **区分执行状态**: 能够识别每次执行的结果
   - `planning_success`: 规划成功
   - `planning_failed`: 规划失败
   - `execution_success`: 执行成功
   - `execution_failed`: 执行失败
   - `running`: 正在执行

3. **关联性**: 能够追溯同一workflow的所有执行记录

4. **前端展示**: 在前端能够直观查看任务的完整执行历史

### 设计原则

- ✅ **零前端改动**: 只需后端通过 `instruction_history` 和 `workmode` 判断，无需前端传递额外字段
- ✅ **简单有效**: 通过少量字段扩展实现需求
- ✅ **向后兼容**: 兼容旧的日志文件

### 3.2 执行阶段识别逻辑

#### 3.2.1 识别规则（后端自动判断）

根据实际代码分析，前端在不同场景下的调用参数：

| 场景 | workmode | instruction_history长度 | stop_after_planner |
|------|----------|------------------------|-------------------|
| 初次规划 | launch | 1 | true |
| 改写规划 | launch | >= 2 | true |
| 重试规划 | launch | >= 2 | true |
| 确认执行 | **production** | >= 1 | false |

**识别逻辑**（优先级从高到低）：

```python
def determine_execution_phase(
    workmode: str, 
    instruction_history: List[str]
) -> ExecutionPhase:
    """
    判断执行阶段
    
    识别逻辑（优先级从高到低）:
    1. 如果 workmode == "production" -> execution（确认执行）
    2. 如果 instruction_history 长度 <= 1 -> initial_planning（初次规划）
    3. 否则 -> re_planning（重规划）
    
    说明：
    - instruction_history 在初次输入时长度为1
    - 改写规划时，会增加新指令，长度变为 >= 2
    - 重试规划时，不会增加指令，但长度仍然是 >= 2
    - 因此通过长度即可区分初次规划和重规划
    """
    if workmode == "production":
        return ExecutionPhase.EXECUTION
    
    if not instruction_history or len(instruction_history) <= 1:
        return ExecutionPhase.INITIAL_PLANNING
    
    return ExecutionPhase.RE_PLANNING
```

**为什么不需要区分"改写"和"重试"？**

根据实际代码分析：

```javascript
// 改写规划：appendHistory=true，会增加instruction_history
runPlannerUpdate(newInstruction, true);

// 重试规划：appendHistory=false，不会增加instruction_history
runPlannerUpdate(lastInstruction, false);
```

虽然两者在前端的处理不同，但对于后端日志记录来说，**都是重新规划**，无需区分具体是改写还是重试。

#### 3.2.2 执行状态识别

在workflow结束时,根据结果更新状态:

```python
def determine_execution_status(
    execution_phase: ExecutionPhase,
    status: str,
    has_planning_steps: bool,
    error: Optional[str]
) -> ExecutionStatus:
    """
    判断执行状态
    """
    if status == "running":
        return ExecutionStatus.RUNNING
    
    if error:
        if execution_phase == ExecutionPhase.EXECUTION:
            return ExecutionStatus.EXECUTION_FAILED
        else:
            return ExecutionStatus.PLANNING_FAILED
    
    if execution_phase == ExecutionPhase.EXECUTION:
        return ExecutionStatus.EXECUTION_SUCCESS
    else:
        return ExecutionStatus.PLANNING_SUCCESS if has_planning_steps else ExecutionStatus.PLANNING_FAILED
```

### 3.3 TaskLogger类扩展

**设计原则**：判断逻辑内聚在TaskLogger中，实现日志记录与主流程的解耦。

#### 3.3.1 新增方法

```python
class TaskLogger:
    # ... 现有方法 ...
    
    def set_execution_phase(self, execution_phase: str) -> None:
        """设置执行阶段"""
        self.execution_phase = execution_phase
    
    def update_execution_status(self, execution_status: str) -> None:
        """更新执行状态"""
        self.execution_status = execution_status
        self._flush()
    
    def to_dict(self) -> Dict[str, Any]:
        """序列化为字典"""
        return {
            "task_id": self.task_id,
            "workflow_id": self.workflow_id,
            "user_query": self.user_query,
            "execution_phase": self.execution_phase,  # 新增
            "execution_status": self.execution_status,  # 新增
            "created_at": self.created_at,
            "finished_at": datetime.now().isoformat() if self.status != "running" else None,
            "status": self.status,
            "history": self.history,
            "error": self.error,
        }

    @staticmethod
    def determine_execution_phase(workmode: str, instruction_history: list) -> str:
        """
        判断执行阶段（静态方法，解耦主流程）

        Args:
            workmode: 工作模式 ("launch" 或 "production")
            instruction_history: 指令历史列表

        Returns:
            执行阶段: "initial_planning" | "re_planning" | "execution"
        """
        # 优先级1: workmode="production" → 确认执行
        if workmode == "production":
            return "execution"

        # 优先级2: instruction_history长度判断
        if len(instruction_history) <= 1:
            return "initial_planning"
        else:
            return "re_planning"

    @staticmethod
    def determine_execution_status(
        execution_phase: str,
        status: str,
        has_planning_steps: bool,
        error: Optional[str] = None
    ) -> str:
        """
        判断执行状态（静态方法，解耦主流程）

        Args:
            execution_phase: 执行阶段
            status: 任务状态 ("running" | "completed" | "failed")
            has_planning_steps: 是否有规划步骤
            error: 错误信息

        Returns:
            执行状态: "planning_success" | "planning_failed" |
                     "execution_success" | "execution_failed" | "running"
        """
        if status == "running":
            return "running"

        if execution_phase in ["initial_planning", "re_planning"]:
            # 规划阶段
            if status == "completed" and has_planning_steps:
                return "planning_success"
            else:
                return "planning_failed"
        else:
            # 执行阶段
            if status == "completed" and not error:
                return "execution_success"
            else:
                return "execution_failed"

    @classmethod
    def load(cls, task_id: str) -> Optional["TaskLogger"]:
        """加载日志（兼容旧日志）"""
        # ... 现有加载逻辑 ...
        # 兼容性处理：如果缺少新字段，设置默认值
        inst.execution_phase = data.get("execution_phase", "initial_planning")
        inst.execution_status = data.get("execution_status", "running")
        # ...
```

## 5. 优点

### 5.1 相比原方案的改进

1. **零前端改动**: 不需要前端传递execution_type字段，完全由后端自动判断
2. **更简单**: 只区分3种阶段（初次规划、重规划、执行），而非4种类型
3. **逻辑更清晰**: 
   - 通过 `workmode` 区分是否执行
   - 通过 `instruction_history` 长度区分初次规划和重规划
4. **向后兼容**: 兼容旧的日志文件，不影响历史数据
5. **可扩展**: 未来可以轻松添加更多状态

### 5.2 核心优势

- ✅ **简单有效**: 通过2个字段即可实现需求
- ✅ **自动识别**: 后端根据现有参数自动判断，无需额外输入
- ✅ **前端友好**: 提供清晰的API接口，便于前端展示
- ✅ **易于维护**: 逻辑简单，代码改动少
- ✅ **解耦设计**: 判断逻辑内聚在TaskLogger中，主流程无需关心细节

### 5.3 架构优势

**判断逻辑内聚在TaskLogger中的好处**：

1. **单一职责原则**：
   - TaskLogger负责日志记录相关的所有逻辑，包括判断执行阶段和状态
   - process.py专注于workflow执行流程，无需了解日志记录的判断细节

2. **易于测试**：
   - 判断逻辑作为静态方法，可以独立单元测试
   - 不依赖workflow运行环境，测试更简单

3. **易于维护**：
   - 判断规则修改时，只需修改TaskLogger类
   - 不会影响主流程代码

4. **代码复用**：
   - 判断逻辑可以在不同地方复用（如日志迁移、数据统计等）
   - 避免重复代码
