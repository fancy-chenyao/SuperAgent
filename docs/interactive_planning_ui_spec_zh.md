# 交互式规划闭环（UI 方案）规格说明

本规格说明整合“整体优化目标 + 最新交互闭环需求”，用于指导 UI 与服务端实现完整的交互式规划与执行闭环。

## 1. 总体目标

1. 统一资源注册与发现：本地 + 多远程 registry。  
2. 统一资源描述与调用：Agent / Tool / Skill / MCP 走同一套元数据与执行路径。  
3. 多服务器支持：资源来自不同服务器且可动态上下线。  
4. 运行时热更新：新增资源无需重启即可发现。  
5. 交互式规划闭环：规划必须停在 Planner 后，用户自然语言补充后重规划，确认后执行。  

## 2. 关键原则

1. Planner 结束后必须暂停，不自动执行。  
2. 用户只通过自然语言补充/修正需求，不接触结构化字段。  
3. 每次补充均基于“全量指令历史”重新规划并回显。  
4. 只有用户确认后才进入执行阶段。  
5. 规划、执行、恢复全链路可追踪。  

## 3. 关键概念

1. 指令历史（Instruction History）  
   - 用户初始输入与所有补充指令的集合。  
   - 每次规划必须基于全量历史。  

2. 计划草案（Plan Draft）  
   - Planner 输出的步骤列表（JSON steps）。  
   - 每轮规划覆盖上一版本。  

3. 执行确认（Execution Confirmation）  
   - 只有用户确认后，才进入执行流程。  

4. 统一资源模型（ResourceSpec）  
   - 统一描述 Agent / Tool / Skill / MCP。  

## 4. 端到端用户流程

1. 用户输入初始需求，提交。  
2. Planner 输出计划并停止。  
3. UI 展示计划草案（步骤卡片）。  
4. 用户自然语言补充/修正。  
5. 基于全量指令历史重规划，回显新计划。  
6. 重复 4-5 直到用户确认执行。  
7. 执行阶段开始并显示实时输出。  
8. 执行失败可恢复。  

## 5. UI 交互设计

### 5.1 Run 页面
1. 输入 User ID 与 Message。  
2. 点击 Run 进入规划阶段。  

### 5.2 计划展示区
1. 展示最新计划草案（步骤卡片）。  
2. 仅展示业务可读信息，不暴露结构化字段。  

### 5.3 自然语言补充区（核心入口）
1. 输入框仅接受自然语言补充。  
2. Apply 后触发“全量历史重规划”。  
3. 规划结果回显。  

### 5.4 执行确认区
1. “确认执行”按钮显式触发执行。  
2. 未确认前禁止进入执行。  

## 6. 服务端能力要求

1. stop_after_planner  关闭自动执行。  
2. instruction_history 维护。  
3. Planner 输出必须为 JSON steps。  
4. 计划回显与版本保存。  
5. 确认执行入口，进入 production。  

## 7. 资源与远程能力协作

1. 远程 Agent / Tool / Skill / MCP 均可进入规划候选。  
2. 规划阶段必须感知远程资源列表。  
3. 执行阶段统一走执行器调度（本地/远程）。  

## 8. 校验规则

1. 计划步骤数 > 0。  
2. Planner 输出可解析为 steps。  
3. 未确认时禁止执行。  
4. 资源缺失需提示或替换。  

## 9. 错误与回退

1. 规划解析失败提示重试。  
2. 规划超时提示重试。  
3. 执行失败支持 checkpoint 恢复。  

## 10. 验收标准

1. Planner 阶段停住，不自动执行。  
2. 自然语言补充触发重规划并回显。  
3. 用户确认才执行。  
4. 远程资源可参与规划与执行。
